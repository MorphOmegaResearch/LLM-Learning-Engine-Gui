  I. Generic Tool Call System Components

  A. Tool Call Detection & Parsing:
  This is the initial stage where the system identifies and interprets a 
  model's intent to use a tool.

   * `interactive.py`:
       * _extract_one_line_json_ENHANCED(raw: str, obj_type: str = None) 
         (Lines ~90-298): The primary function for detecting tool calls. 
         It employs multiple strategies:
           * Attempts FormatTranslator for Python-style function calls.
           * Parses clean JSON lines (OpenCode, Gemma-tools, Function 
             call, Direct tool formats).
           * Uses regex patterns to find embedded JSON.
           * Includes fallback detection for natural language patterns 
             (e.g., "read file 'x'").
       * _extract_tool_calls_from_response(response: str) (Lines ~70-85): 
         Extracts the first valid tool call JSON object, handling 
         concatenated JSONs.
   * `langchain_adapter_simple.py`:
       * SimpleLangChainAdapter.parse_and_validate(model_output: str) 
         (Lines ~40-60): Offers multiple parsing strategies:
           * _try_direct_json(text: str) (Lines ~62-75): Direct JSON 
             parsing.
           * _try_partial_json(text: str) (Lines ~77-90): Partial JSON 
             parsing using json_fixer.smart_json_parse.
           * Falls back to 
             FormatTranslator.translate_tool_call(model_output).
       * _normalize_tool_call(parsed: Dict) (Lines ~92-100): Standardizes 
         parsed tool calls into the OpenCode format 
         ({"type":"tool_call","name":"TOOL_NAME","args":{}}).

  B. Validation:
  Ensures tool calls are well-formed, safe, and adhere to policies before 
  execution.

   * `tool_orchestrator.py`:
       * validate_tool_call(tool_name: str, args: Dict[str, Any]) (Lines 
         ~743-900): A comprehensive validation method that:
           * Verifies tool existence and enablement.
           * Suggests corrections for common tool name mistakes (e.g., ls 
             to bash_execute).
           * Determines the risk_level for the operation.
           * Performs tool-specific argument validations (e.g., file_write
              for overwrite risk, bash_execute for dangerous commands).
           * Determines if user confirmation_required based on risk and 
             configuration.
   * Individual Tool Classes (`tools.py`):
       * BaseTool._validate_path(file_path: str) (Lines ~60-70): Confirms 
         paths are within the working directory.
       * BaseTool._safe_path(file_path: str) (Lines ~72-78): Returns a 
         validated Path object.
       * Each tool's execute method often includes specific parameter 
         validation.
   * `command_policy.py` (used by `BashExecuteTool` and 
     `GitOperationsTool`):
       * CommandPolicy.parse_command_safe(command: str): Parses and 
         validates bash commands against an allow-list.

  C. Parameter Translation (Alias System):
  Maps varied tool names and arguments from model training data to 
  OpenCode's expected format.

   * `tool_alias_translator.py`:
       * ToolAliasTranslator class (Lines ~15-150): Contains self.aliases 
         mapping aliases to OpenCode tools and argument transformer 
         functions.
       * translate(self, tool_name: str, args: Dict[str, Any]) (Lines 
         ~50-65): The core method for translation.
       * _map_file_path, _map_file_write, _map_python_code, etc. (Lines 
         ~100-150): Functions to transform arguments for specific tool 
         types.
   * `interactive.py`:
       * _safe_execute_tool(interactive_self, name: str, args: dict) 
         (Lines ~300-340): Contains in-line parameter normalization logic 
         for file operations, file_search, grep_search, and directory_list
          before calling the tool manager. This appears to be a separate 
         normalization layer.

  D. Confirmation Gates:
  A critical safety mechanism to ensure user approval for potentially 
  risky operations.

   * `tool_orchestrator.py`:
       * ToolRiskLevel Enum (Lines ~25-30): Defines risk levels (SAFE, 
         LOW, MEDIUM, HIGH, CRITICAL).
       * ConfirmationGate Enum (Lines ~32-36): Defines confirmation types 
         (NONE, IMPLICIT, EXPLICIT, MANDATORY).
       * self.risk_profiles (Lines ~100-130): Maps tool names to 
         ToolRiskLevel.
       * self.confirmation_rules (Lines ~132-138): Maps ToolRiskLevel to 
         ConfirmationGate.
       * _get_user_confirmation(self, chain: ToolChain) (Lines ~555-570): 
         Prompts the user for confirmation based on the chain's risk 
         score.
   * `tools.py`:
       * confirmation_gates.py (imported): Used by tools like 
         FileWriteTool and BashExecuteTool for specific action 
         confirmations (e.g., overwriting, dangerous commands).
       * ThinkTimeTool.execute() (Lines ~1096-1120): Includes its own 
         Confirm.ask() for user approval.

  E. Execution Flow:
  The process of running tools, including retries and resource 
  management.

   * `tool_orchestrator.py`:
       * execute_tool_chain(self, chain: ToolChain) (Lines ~460-530): The 
         main entry point for executing a chain of tools.
       * _execute_sequential(self, chain: ToolChain, result: 
         OrchestrationResult, time_budget: TimeBudget) (Lines ~572-620): 
         Executes tools one by one.
       * _execute_parallel(self, chain: ToolChain, result: 
         OrchestrationResult, time_budget: TimeBudget) (Lines ~622-650): 
         Executes tools in parallel where possible.
       * _execute_single_operation(self, operation: ToolOperation) (Lines 
         ~670-690): Calls the ToolManager to execute a single tool.
   * `tools.py`:
       * ToolManager.execute_tool(self, tool_name: str, **kwargs) (Lines 
         ~1560-1630): This method:
           * Checks tool enablement and existence.
           * Applies argument normalization.
           * Handles caching for read-only operations.
           * Implements retry logic (max_retries).
           * Calls the individual tool's execute method.
           * Special handling for change_directory to propagate the new 
             working directory.
   * `interactive.py`:
       * _safe_execute_tool(interactive_self, name: str, args: dict) 
         (Lines ~300-340): Wraps tool_manager.execute_tool with a rich 
         spinner for live progress.

  F. Chaining (Tool Chains & Auto-Chains):
  Mechanisms for executing multiple tools in sequence or in response to 
  previous outputs.

   * `tool_orchestrator.py`:
       * ToolChain dataclass (Lines ~40-47): Represents a sequence of 
         operations.
       * analyze_tool_request(self, user_input: str, route_metadata: Dict)
          (Lines ~150-160): Plans optimal tool chains from user input.
       * _create_optimized_chain(self, operations: List[ToolOperation], 
         context: str) (Lines ~290-310): Optimizes operation order.
       * _execute_sequential and _execute_parallel: Handle chain 
         execution.
   * `config.yaml`:
       * system_prompt section (Lines ~117-120): Defines CHAINING POLICY 
         for auto-executed safe chains (file_search → file_read, 
         directory_list → file_read).
   * `interactive.py`:
       * _call_model_with_tool_result (Lines ~342-390): Feeds tool results
          back to the model, enabling implicit chaining.
       * _get_dynamic_system_prompt (Lines ~490-500): Informs the model 
         about the auto-chaining policy.

  G. Result Formatting:
  How tool outputs are presented to the model and user for clarity and 
  actionability.

   * `interactive.py`:
       * _format_tool_result_as_text(tool_name: str, tool_args: dict, 
         result: dict, step_num: int = None) (Lines ~400-500): Formats 
         results as plain text for model readability, including status, 
         output, and next action hints. This is the primary method for 
         model communication.
       * _format_tool_result_for_model(tool_name: str, tool_args: dict, 
         result: dict) (Lines ~502-700): Returns a dictionary for 
         "backward compatibility," providing detailed error summaries and 
         next action hints.
