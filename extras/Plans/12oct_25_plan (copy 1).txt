  Phase 1: Unify Tool & Schema Management

  Objective: Execute the Tool_management_plan.txt. This is the highest 
  priority as it standardizes the tool system, which all subsequent 
  phases will depend on.

   * Step 1.1: Centralize Backend Assets.
       * Action: Create the new central directory: 
         Training_Data-Sets/Schema_Profiles/.
       * Action: Move the existing profile files from 
         Data/tabs/custom_code_tab/tool_schemas_configs/ to this new 
         directory.
       * Action: Solidify Training_Data-Sets/Schemas/ as the single source
          for all individual tool definitions.
       * Considerations: This is primarily a file system operation. I must
          ensure that any code referencing the old tool_schemas_configs 
         path is updated to point to the new Schema_Profiles directory to 
         prevent breaking the chat tab's functionality.

   * Step 1.2: Overhaul the "Tools" Sub-Tab into a Profile Manager.
       * Action: Redesign the UI in tools_tab.py to feature a dropdown 
         menu for selecting, creating, and deleting profiles.
       * Action: The tool checkboxes will now dynamically update to 
         reflect the contents of the profile currently selected in the 
         dropdown.
       * Action: The "Save Settings" button will be repurposed to "Save 
         Profile."
       * Considerations: This is a significant UI modification. The 
         internal logic of tools_tab.py will need to change from managing 
         a single global state to handling multiple, named profile states 
         that are loaded from and saved to different files.

   * Step 1.3: Integrate the Unified Profile System.
       * Action: Update the "Chat" sub-tab (chat_interface_tab.py) to 
         populate its "Tool Schema" selector from the new 
         Training_Data-Sets/Schema_Profiles/ directory.
       * Action: Update the "Training" tab's Runner Panel 
         (runner_panel.py) so its "Evaluation Context" dropdown also lists
          and uses these new profiles.
       * Considerations: This ensures a consistent user experience. When 
         you create a profile in the "Tools" tab, it should immediately be
          available for use in both the chat and evaluation contexts.

   * Step 1.4: Implement Context-Aware Data Collection.
       * Action: Modify the ToolCallLogger in tool_call_logger.py to 
         accept a context dictionary in its logging methods.
       * Action: Update the chat_interface_tab.py to capture the active 
         tool profile name and pass it to the logger with every logged 
         tool call.
       * Considerations: The format of the log files 
         (tool_realtime_data.jsonl and tool_training_data.jsonl) will 
         change. I must verify that downstream systems, like the analytics
          functions for the "Skills" tab, are either compatible with this 
         new format or are updated accordingly.

  Phase 3: Implement Model Gamification & Evolution

  Objective: Implement the gamification system described in gamify.txt to 
  provide clear, motivating feedback on model progression.

   * Step 3.1: Implement Backend Logic & Data Structures.
       * Action: Create the new Python files model_typing.py (to define 
         model "Types") and tiering.py (to calculate ranks and "Evolution 
         Points").
       * Action: Establish the skills/<model_id>.json file structure to 
         store a model's type, rank, and capability scores.
       * Considerations: The logic in these new files will be critical and
          must be robust. The calculations for rank and type will directly
          depend on the structure of the evaluation reports.

   * Step 3.2: Integrate Gamification with the Evaluation Engine.
       * Action: Modify evaluation_engine.py to call the new tiering and 
         typing logic after each evaluation run is complete.
       * Considerations: This creates a dependency between the evaluation 
         engine and the new gamification logic. I need to ensure they 
         communicate correctly and that failures in the gamification step 
         don't prevent evaluation reports from being saved.

   * Step 3.3: Build the Gamification UI.
       * Action: In models_tab.py, modify the UI to display the "Type" and
          "Rank" as colored badges next to the model name.
       * Action: The "Skills" tab will be updated to show the detailed 
         capability scores and the requirements for reaching the next 
         tier.
       * Considerations: This involves creating or modifying custom 
         widgets to display the new gamified information in a visually 
         appealing way.

   * Step 3.4: Implement the "Evolution Conformer".
       * Action: Create a new UI pop-up that appears when a model's 
         evaluation scores meet the criteria for an "Evolution."
       * Action: This pop-up will present the user with choices like 
         "Promote to New Base Model" or "Unlock Growth Mode."
       * Considerations: This is a complex UI component that is 
         conditionally triggered. The actions it performs (like merging 
         adapters or enabling full-weight training) are significant and 
         must be handled carefully, likely by calling other scripts.
