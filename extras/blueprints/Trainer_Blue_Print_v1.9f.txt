  ## KEY ARCHITECTURAL CHANGES IN v1.9f

  ### Mode System Architecture

  MODE_ADVANCED_MAPPINGS defines which advanced systems are enabled per mode:
  - Standard: No advanced systems (clean baseline)
  - Fast: 3 systems (json_repair, schema_validation, model_selector)
  - Smart: 12 systems (includes orchestrator, routing, context scoring)
  - Think: 30 systems (all advanced features enabled)

  Mode parameters are now fully configurable per mode with controls:
  - Resource Usage: Dropdown (Default, 25%, 50%, 75%)
  - Max Context Tokens: ◀ [value] ▶ (1024-16384, step: 512)
  - Generation Step: ◀ [value] ▶ tokens (8-128, step: 8)
  - Token Caps: ◀ [value] ▶ with separate limits per type
  - CPU Threads: Dropdown (Default, Minimal, Half, Most)
  - Quality Mode: Dropdown (Standard, Fast, Auto/Smart, Think)

  ### Tool Execution Architecture (Corrected)

  The system does NOT use Ollama's native tools API. Instead:

  1. Tool schemas embedded in system prompts
  2. Model outputs tool calls as text
  3. Format Translator parses text to detect tool usage
  4. Tools tab settings checked before execution
  5. Only enabled tools are executed

  This approach:
  - Works with any model (not just Ollama-tools-compatible)
  - Provides fine-grained control via Tools tab
  - Maintains safety through enable/disable switches
  - Functions in Standard mode (without Format Translator) if model outputs proper format

  ### Chat Interface Layout (Reorganized)

  Top Bar:
  - Active Model indicator (left)
  - Tool Schema, System Prompt, Change Dir, ⚡ Mode buttons (center/right)

  Chat Display Area:
  - Scrollable chat history
  - Auto-scroll to bottom on new messages

  Button Bar (Above Input):

  - Top-right hosts 📌 Mount | 📍 Dismount | 🆕 New Chat | 🗑 Delete Chat

  Input Area:
  - Text input (left)
  - Send/Stop buttons (right)
  - Instructions: "Press Enter to send • Shift+Enter for new line"
  - Left of input: ⚙ Quick Actions and indicator strip (⏱, 📂, 🔧, 🗒, ⚡, 📝, 👁, 🧠, 💾)

  ### Conversations Sidebar (Categorized)
  - Replaced flat list with categorized Treeview: Latest (Mon..Sun with Today first, limited to 15), Week, Month, Year
  - Single-click shows a Quick View popup near the list with:
    • Model header color‑coded with <Class> (falls back to <Unassigned>)

  - Double‑click opens the selected chat

  ### Show Thoughts (Quick Action)
  - 👁 toggle streams incremental output into the chat as grey italic “thoughts” while awaiting the final answer
  - Final response remains parsed for tools and saved normally

  ### RAG Controls
  - Chat: per‑chat 🧠 RAG (default OFF) and 💾 Save RAG per session (default OFF)
  - If Save is ON and RAG changed, user is prompted on New Chat/opening another chat to persist the new setting to session metadata

  ### Create Project (Quick Action)
  - 🗂 “Create Project from this chat” prompts for a project name, saves current chat and metadata to Data/projects/<name>, switches to Projects tab, and selects it

  ### Projects Tab (NEW)

  - Default RAG ON for project chats; indicators include 🧠 and 🗂 (Project: name)
  - Auto‑project creation (optional) when chatting in Projects with no project selected

  ```json
    "current_mode": "smart",
    "mode_parameters": {
      "standard": { /* parameters */ },
      "fast": { /* parameters */ },
      "smart": { /* parameters */ },
      "think": { /* parameters */ }

    "think_time_max_seconds": 300
  ```

  ```json
      "grep_search": true,

  ```

  ## COMPONENT INTERACTIONS

  ### Mode Change Flow

  1. User selects mode (Mode tab radio button OR Chat tab ⚡ Mode dialog)
  2. on_mode_changed() called
  3. MODE_ADVANCED_MAPPINGS consulted
  4. Advanced systems enabled/disabled per mode
  5. Settings tab notified via on_mode_changed()
  6. Advanced tab displays active mode
  7. Mode-controlled systems show 🔒 and are disabled
  8. Chat interface receives set_mode_parameters()
  9. System initializes advanced components if not in Standard mode

  ### Tool Execution Flow

  1. User sends message to model
  2. Model response parsed (JSON)
  3. Format Translator detects tool calls in text (if enabled)
  4. handle_tool_calls() invoked
  5. Tools tab settings checked via get_enabled_tools()
  6. Disabled tools skipped with error message
  7. Enabled tools executed via tool_executor
  8. Tool Orchestrator applies risk gates (if enabled)
  9. Results displayed in chat
  10. Follow-up generation if needed

  ### Chat History Flow

  1. Every model response triggers _auto_save_conversation()
  2. ChatHistoryManager.save_conversation() called
  3. Session includes: chat_history, model, mode, temperature, prompts, schemas
  4. Saved to Data/chat_history/{session_id}.json
  5. History tab displays all sessions via list_conversations()
  6. Load button restores full conversation state
  7. Clear Chat saves current session before clearing

GGUF Export for Variants

## v1.9g — Quick Actions, Per‑Chat Tools, Pane Locks, Auto‑Training Loop

Overview
- Adds Quick Actions gear in Chat (bottom‑left), per‑chat Tools overrides with scrollable UI, movable/lockable side panes (left Conversations and right Models), a fully hands‑free Auto‑Training loop from Custom Code, and improved model mounting + Active Model visuals.

UI: Quick Actions (⚙️)
- Main popup (compact, anchored above the gear button):
  - 📂 Change Working Directory
  - 🔧 Manage Tools → opens Tools view
  - 🏋️ Training Mode button with colored state (green ON, red OFF)
- Tools view (enlarged popup, scrollable):
  - Lists all Tools from the Tools tab; checkboxes pre‑populated from Tools tab selections
  - Save persists a per‑chat override (does not change system defaults)
  - Hover tooltips show descriptions; Back (⬅) returns to compact view
  - On Save, Chat logs a diff summary (Enabled/Disabled; changes vs defaults)

Per‑Chat Tool Overrides
- get_tool_schemas() prefers session overrides over backend defaults
- Chat auto‑save persists session_tools in conversation metadata
- Loading a conversation restores session_tools automatically

Side Pane Locks + Defaults
- Left Conversations and Right Models are resizable via sash and have lock buttons (🔒/🔓)
- Clicking lock captures the current width, enforces on resize, and persists to custom_code_settings.json
- On launch, both panes start locked using the last saved widths (serve as defaults)
- While locked, drag/hover resizing is disabled and cursors show as non‑resizable

Active Model + Mounting
- Active Model label colors by class (novice green, skilled blue, expert purple, master orange, artifact red)
- Mount button shows red (not mounted) → green (mounted)
- Mount resolves the model from internal state; if missing, falls back to the Active Model label
- Preflight checks report clear errors (no model selected, ollama missing, stderr/stdout on failure)

Auto‑Training Loop (Custom Code → Training → Models)
- When Training Mode is ON:
  - After tool calls, failed/refusal cases produce strict JSONL in Tools/ with scenario tags
  - Training tab selects the dataset automatically and saves the Training Profile
  - If auto‑start training is ON (Training tab → Settings): training begins immediately
  - Runner emits start/progress/complete events; a popup shows progress; “View Logs” focuses Runner
  - On completion, export + re‑eval can run automatically (Models tab handler); fallback strict context used if necessary
  - Promotion remains threshold‑gated

Notes for Blueprint consumers
- Quick Actions centralizes common actions; we removed the top‑bar Change Dir and moved Training toggle into the popup
- Tools checked in Quick Actions override the system/type defaults for that chat only
- Pane locks define your launch defaults; use lock→unlock to set and test widths, then relaunch locked

- Apply Type Plan → profiles saved; Collections updated under Type.
- Variant selection → Evaluation target set; novice suite auto‑selected; baseline auto‑toggled if fresh.
- GGUF export → .gguf written; tag created under Assigned when Ollama available or auto‑started.
- Skills/Baselines/Compare refresh on run.

  ## TECHNICAL NOTES

  ### BaseTab.bind_mousewheel_to_canvas()
  Universal method for all tabs to enable mousewheel scrolling.
  Handles platform differences transparently.

  ### MODE_ADVANCED_MAPPINGS
  Central source of truth for mode-to-system relationships.
  Located in advanced_settings_tab.py lines 57-133.

  ### Tool Prompt-Based Approach
  Critical architectural decision: NO use of payload["tools"].
  Tool schemas embedded in system prompts instead.
  Enables universal model compatibility.

  ### Session Metadata
  Comprehensive snapshot of configuration:
  - mode, temperature
  - system_prompt, tool_schema
  - working_directory
  - training_mode_enabled
  - model_name
  - tool_settings (which tools enabled/disabled)
